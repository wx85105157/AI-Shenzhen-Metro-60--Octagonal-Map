# 深圳 60° 八向轨道交通交互示意图：工作计划

> 项目：`wx85105157/AI-Shenzhen-Metro-60--Octagonal-Map`  
> 需求确认日期：2026-07-22  
> 首版数据快照目标日期：2026-07-22

## 1. 项目目标

创建一个无需后端、可部署到 GitHub Pages 的静态交互式 HTML/SVG 轨道交通示意图。视觉语言以用户提供的“60° 八向走线布局”参考图为依据；深圳为主体，并按真实资料维护运营、在建和规划轨道交通数据。

首版强调：

- SVG 清晰渲染、缩放、拖拽与图层控制；
- 真实数据与渲染/交互代码分离；
- 用真实经纬度和官方资料作为底层依据；
- 在必要时以八方向示意坐标改善可读性；
- 可将参考图作为可调透明度的校验叠加层；
- 数据来源可追溯，绝不估算或虚构首末班、计划开通日期等信息。

## 2. 首版地理与线路范围

### 2.1 深圳主体范围

- 深圳市内运营、在建、规划的城市轨道交通按官方资料逐步录入。
- 图例与数据模型预留：地铁、城际/国铁、其他线路三个分类。
- 首个可运行版本可先展示一条真实运营线路作为数据及渲染样例；其他线路将在后续阶段逐条核验并增加。
- 可纳入深圳地铁、支线、机场快线属性线路、云巴、有轨电车等，但必须先完成官方资料核验。

### 2.2 东莞与惠州方向

- 不绘制东莞、惠州完整的市内轨道网络。
- 仅保留深圳直接衔接的跨市轨道/城际走向和必要边界侧节点或终点说明。
- 在跨出深圳区域后，可使用“至东莞/惠州方向”的终点表达；具体站点仅在具有官方资料、且对衔接表达确有必要时加入。

### 2.3 香港方向

- 不绘制香港境内 MTR 线路与站点。
- 不在图中绘制香港西九龙站站点。
- 广深港高铁跨越香港边境后，以终点文字 `to 香港西九龙站` 标注。

### 2.4 国铁、城际与普速铁路

- 普速铁路不列入首版。
- 国铁与城际的完整网络、站点与换乘关系列为后续开发计划。
- 首版只在必要时预留分类、数据接口与跨境/跨市方向表达。

## 3. 状态分类与显示规则

所有线路和站点的业务状态独立存储，至少包括以下层级：

| 状态 | 数据含义 | 默认视觉 |
| --- | --- | --- |
| `operating` | 已投入运营 | 对应线路颜色的实线 |
| `under_construction` | 已有官方建设依据、处于建设阶段 | 对应线路颜色的虚线 |
| `planned_approved` | 已批复或列入近期建设规划、但尚未开工 | 灰色线（可筛选） |
| `planned_official` | 已纳入官方规划、但未获批或建设条件未明确 | 灰色线（较低透明度，可筛选） |
| `vision_or_study` | 远景、研究或方案阶段 | 灰色线（最低透明度/独立筛选） |

- 规划各级数据分级存储，并可在界面中分别筛选。
- 在建线路和站点显示官方公布的计划开通时间。
- 若只有年份、季度、或未公布，则严格按官方原文精度显示，例如“预计 2028 年”或“官方未公布”。
- 不得根据新闻推测、施工进度或第三方信息补全日期。

## 4. 数据与来源原则

### 4.1 官方来源优先

最终数据必须能够追溯到官方发布材料，例如：

- 轨道运营单位；
- 项目业主；
- 政府部门、交通主管部门；
- 发展改革、规划和自然资源主管部门；
- 政府公报、法定规划、建设批复、官方新闻稿或公告。

非官方地图、论坛、百科、社交媒体仅可用于发现线索，不能作为最终事实来源。

### 4.2 每项数据的溯源字段

线路、站点、建设状态与服务时刻数据均应尽可能包含：

```json
{
  "sourceName": "发布机构或资料名称",
  "sourceUrl": "官方来源地址",
  "publishedAt": "YYYY-MM-DD 或原始发布日期文本",
  "checkedAt": "2026-07-22",
  "notes": "适用范围、例外或核验说明"
}
```

### 4.3 首末班与方向信息

- 仅展示运营方公开的车站/方向级常规首末班时刻。
- 若官方只公开线路级信息，或未公开某站某方向时刻，则展示“官方未公开”，不估算。
- 悬停运营站点时，显示：中英文名称、所属线路、换乘线路、方向、首班、末班、官方来源、数据更新时间及备注。
- 工作日/周末或节假日差异应按官方资料明确标注；首版优先采用常规日资料。

### 4.4 名称规则

- 默认显示简体中文站名与官方英文/拼音副标题。
- 架构预留中英文切换能力。
- 在建及规划站使用官方名称；如果官方明确为暂定名，显示“暂定名”标记。

## 5. 坐标、拓扑与制图规则

### 5.1 双坐标模型

每个可定位对象应分开保存：

- `geoPosition`：真实经纬度或可追溯的地理位置；
- `schematicPosition`：用于八向示意图的 SVG 坐标；
- `labelPosition`：站名标签位置与避让参数。

### 5.2 优先级

1. 真实经纬度、行政边界、公园边界作为基础地理数据保存；
2. 当真实空间关系与参考图比例接近且可读时，优先根据地理坐标投影生成布局；
3. 当真实方位、文字避让或线路交叉影响可读性时，仅调整 `schematicPosition`，使折线遵循 0°、45°、90°、135° 等八向规则；
4. 站点走向、相对间距和整体布局应尽量贴近参考图，同时明确其为拓扑示意而非测绘地图。

## 6. 城市边界与公园图层

- 城市边界：仅展示为适配八向布局而简化的城市边界，不以区县边界为首版目标。
- 公园：仅展示参考图例所示的大型公园，不额外新增小公园。
- 城市名称、公园名称、边界几何均作为独立数据类型存储。
- 后续计划增加专项核验：排查是否出现新的大型公园、既有大型公园是否更名，以及边界资料是否更新。

## 7. 参考图叠加

用户确认可将参考图公开提交至本 Git 仓库，作为校验素材。

实现要求：

- 默认关闭；
- 支持开关；
- 支持透明度滑杆；
- 与 SVG 视图共用缩放与拖拽变换，以便对照站位、线路走向和间距；
- 页面中说明其用途为“布局校验参考层”。

## 8. 技术方案

- 纯前端静态实现：HTML、CSS、JavaScript、SVG、JSON/GeoJSON；
- 无后端依赖；
- 可部署至 GitHub Pages；
- SVG 作为路线、站点、标签和边界的主渲染层；
- SVG 根视图与参考图叠加层共享同一平移/缩放坐标系统；
- 提供鼠标滚轮/触控板缩放、拖拽平移、重置视图、图层开关和状态筛选；
- 数据、样式、渲染逻辑和交互逻辑分目录维护。

建议目录：

```text
/
├── index.html
├── README.md
├── docs/
│   ├── WORK_PLAN.md
│   ├── DATA_SPEC.md
│   └── SOURCES.md
├── assets/
│   └── reference/                 # 用户确认可公开的参考图
├── src/
│   ├── app.js
│   ├── render/
│   ├── interaction/
│   ├── styles/
│   └── data/
│       ├── lines.json
│       ├── stations.json
│       ├── services.json
│       ├── projects.json
│       ├── cities.geojson
│       └── parks.geojson
└── tests/
```

## 9. 数据模型分层

| 数据文件 | 责任 |
| --- | --- |
| `lines.json` | 线路编号、名称、类别、颜色、状态、示意路径、来源 |
| `stations.json` | 站名、双语名、真实/示意坐标、状态、换乘、来源 |
| `services.json` | 运营站点的方向、首末班、适用日、来源 |
| `projects.json` | 在建/规划项目、层级、计划开通时间、来源 |
| `cities.geojson` | 城市边界及名称定位 |
| `parks.geojson` | 大型公园边界及名称定位 |

业务数据不得写死在渲染与交互代码中。

## 10. 分阶段实施与验收

每一阶段完成后先进行运行与基础校验，等待确认后再继续下一阶段。

### 阶段 1：项目骨架与交互底座

工作内容：

1. 初始化静态项目与目录结构；
2. 建立数据 Schema 和示例数据；
3. 建立 SVG 画布；
4. 实现缩放、拖拽、重置视图；
5. 实现图层控制面板与状态筛选框架；
6. 接入本地参考图叠加层和透明度控制；
7. 增加一条真实运营线路的最小展示样例；
8. 添加基础数据校验与本地运行说明。

验收重点：

- 缩放后，SVG 与参考图叠加层同步缩放和平移；
- 拖拽不会导致两层产生相对偏移；
- 可关闭/打开参考层并调整透明度；
- 数据文件与交互代码完全分离；
- 示例线路可正常渲染与悬停。

### 阶段 2：运营线路与站点数据

工作内容：

1. 逐条核验深圳已运营轨道线路；
2. 录入线路、站点、换乘关系、真实地理坐标和示意坐标；
3. 依据参考图与地理关系微调八向走线；
4. 补充官方来源索引。

验收重点：线路拓扑、站点顺序、换乘关系、来源链接和相对布局可核验。

### 阶段 3：运营服务信息交互

工作内容：

1. 核验并录入可获得的官方首末班与方向资料；
2. 完善运营站点悬停卡片；
3. 对未公开数据明确展示“官方未公开”。

验收重点：任何展示时刻均可溯源；无来源时不显示猜测值。

### 阶段 4：在建与规划网络

工作内容：

1. 录入已开工项目；
2. 按官方依据录入已批复、官方规划、远景/研究项目；
3. 使用虚线、灰线和筛选项呈现不同状态；
4. 录入官方计划开通时间或“未公布”。

验收重点：状态、规划层级与开通时间均有清晰来源和不确定性表达。

### 阶段 5：跨市/跨境方向与铁路扩展

工作内容：

1. 增加深圳至东莞、惠州方向的必要跨市衔接表达；
2. 增加至香港边界的衔接表达；
3. 广深港高铁跨境后以 `to 香港西九龙站` 作为终点文字；
4. 按后续确认范围添加国铁和城际线路；不添加普速铁路。

验收重点：不展开东莞、惠州或香港境内完整城市轨道网；跨境终点表达准确。

### 阶段 6：城市边界与大型公园

工作内容：

1. 添加简化城市边界和城市名称；
2. 添加参考图所列大型公园与边界；
3. 调整图层层级、标签和视觉权重；
4. 建立后续“大型公园新增/更名核验”清单。

验收重点：边界与标签不干扰交通网络，并保留独立可维护数据。

### 阶段 7：核验、视觉调优与发布

工作内容：

1. 数据来源复查；
2. 对参考图叠加进行站位和比例校验；
3. 检查不同缩放级别、屏幕尺寸和交互状态；
4. 完善 README、数据更新流程和 GitHub Pages 部署说明。

验收重点：项目可直接运行、可发布、可维护，且数据更新流程清楚。

## 11. 已确认的非阻塞约束

- 不使用后端；
- 参考图可以公开提交；
- 每阶段调试通过、经确认后才进入下一阶段；
- 所有数据以 2026-07-22 为首版快照目标；
- 后续变更必须记录对应数据来源、核验日期和影响范围。

## 12. 后续变更记录规范

每次更新数据时应记录：

- 更新日期；
- 修改的数据文件；
- 线路/站点/边界对象；
- 官方来源；
- 原因（开通、延期、命名变化、规划调整、边界或公园更新等）；
- 是否影响示意坐标或视觉布局。
